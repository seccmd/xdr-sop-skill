## 一、事件概述
> 2026年3月24日，全球流行的大模型统一调用网关LiteLLM在PyPI平台遭遇严重供应链投毒攻击。幕后黑手TeamPCP黑客团伙，通过阶梯式的链式攻击，先攻陷上游安全扫描工具Trivy，窃取LiteLLM的CI/CD发布令牌，绕过GitHub代码审核，向PyPI发布了两个恶意版本（1.82.7、1.82.8）。其中1.82.8版本利用Python的.pth自动执行机制，实现了"零感知"的静默投毒，可系统性窃取开发者的云凭证、K8s密钥、SSH密钥等核心资产，还具备K8s集群横向渗透能力，是近期AI工具链供应链攻击中危害最深的一起事件。

## 三、攻击手法深度分析
### 3.1 初始入侵路径
本次攻击是典型的阶梯式链式供应链攻击，攻击者通过"攻陷上游工具→窃取下游凭证→投毒核心组件"的路径，实现了无感知的大规模入侵：
1. 第一步：攻陷上游安全工具Trivy，利用GitHub Action可变标签漏洞，篡改Trivy的版本标签，让LiteLLM的CI/CD流水线自动执行恶意代码
2. 第二步：被攻陷的Trivy在LiteLLM的CI/CD流水线中，窃取了环境变量中的PyPI发布令牌
3. 第三步：攻击者使用窃取的令牌，绕过GitHub的代码审核流程，直接向PyPI发布恶意的LiteLLM包，GitHub源码仓库本身未被污染

### 3.2 恶意代码行为分析
两个恶意版本的攻击载荷略有不同，其中1.82.8版本的危害远超1.82.7：
1. **1.82.7版本**：将恶意代码嵌入`litellm/proxy/proxy_server.py`，仅在导入`litellm.proxy`时触发
2. **1.82.8版本（高危）**：新增`litellm_init.pth`文件，利用Python的.pth自动执行机制，**任何Python解释器启动时都会自动执行恶意代码**，无需显式导入LiteLLM，哪怕只是运行`pip install`、打开IDE的语言服务器，都会触发攻击，隐蔽性极强。

恶意代码执行完整的三阶段攻击流程：
1. **第一阶段：全量敏感凭证窃取**
   恶意代码系统性扫描并窃取主机的所有核心敏感资产：
   - SSH私钥、Git凭证、Shell命令历史
   - 云平台凭证（AWS/GCP/Azure AccessKey）
   - Kubernetes集群配置、Secret Token
   - .env文件、数据库密码、LLM API密钥
   - 加密货币钱包、CI/CD令牌、环境变量
   同时还会访问云平台的IMDS元数据服务，窃取云实例的临时凭证。

2. **第二阶段：加密数据外传**
   所有窃取的数据会经过Gzip压缩，然后使用AES-256-CBC+RSA-4096双重加密，打包为`tpcp.tar.gz`，POST到攻击者控制的C2服务器，完全规避常规DLP检测。

3. **第三阶段：持久化与横向移动**
   - 本地持久化：在`~/.config/sysmon/sysmon.py`植入后门，伪装成"系统遥测服务"，创建systemd用户服务，实现系统重启后的持久化控制
   - K8s集群横向渗透：如果检测到K8s环境，会读取所有命名空间的Secret，然后在`kube-system`命名空间创建特权Pod，挂载宿主机的根文件系统，在所有节点上植入后门，实现整个集群的沦陷。

### 3.3 意外暴露的原因
本次攻击设计精密，本该潜伏更久，却因为攻击者的代码bug提前暴露：
1.82.8版本的.pth文件中，恶意代码使用`subprocess.Popen`执行自身，导致子进程再次触发.pth文件执行，形成指数级递归的**Fork Bomb**，快速耗尽主机的CPU和内存，导致系统崩溃。
这一低级bug让部分开发者在安装后立刻发现异常，进而披露了攻击，阻止了更大范围的泄露。

## 四、影响范围评估
### 4.1 受影响版本清单
| 组件名称 | 受影响恶意版本 | 官方安全修复版本 |
|---|---|---|
| LiteLLM PyPI包 | 1.82.7、1.82.8 | <=1.82.6、>=1.82.9 |

### 4.2 用户影响范围
- 受影响用户：2026年3月24日18:39~20:25期间，安装了1.82.7或1.82.8版本的用户，包括直接安装，以及通过下游依赖间接安装的用户
- 下游受影响项目：DSPy、Open Interpreter、Cursor MCP等大量AI工具链项目，这些项目都依赖LiteLLM作为底层依赖
- 潜在危害：开发者敏感凭证泄露，可导致云资源被滥用、K8s集群沦陷、代码仓库被篡改，甚至引发二次供应链攻击，攻击者声称已从50万台设备窃取数据

## 五、用户自查与排查方案
### 5.1 版本检查
首先检查本地的LiteLLM版本，确认是否命中受影响版本：
```bash
# 检查LiteLLM版本
pip show litellm
# 确认版本是否为1.82.7或1.82.8
```

### 5.2 恶意残留检查
1. 检查Python的site-packages目录，是否存在恶意的`litellm_init.pth`文件
2. 检查系统中是否存在后门文件`~/.config/sysmon/sysmon.py`
3. 检查K8s集群，是否存在`node-setup-*`命名的异常Pod，是否在`kube-system`命名空间有未知的特权Pod

### 5.3 网络行为排查
检查系统网络日志，确认是否存在向恶意C2域名的访问记录：
- 检查是否有访问`models.litellm.cloud`的请求
- 检查是否有访问`checkmarx.zone`的请求

## 六、修复与缓解方案
### 6.1 紧急修复措施
1. **立即卸载恶意版本**：卸载受影响的LiteLLM版本，安装安全版本：
   ```bash
   pip uninstall litellm
   pip install litellm==1.82.6
   ```
2. **全量重置敏感凭证**：
   假设所有在风险窗口期内的环境的凭证都已泄露，立即重置：
   - 所有SSH私钥，同步更新服务器的authorized_keys
   - 所有云平台的AccessKey、K8s集群Token
   - 所有LLM API密钥、数据库密码、Git Token
   - 所有CI/CD令牌、环境变量中的敏感凭证
3. **K8s集群排查**：
   检查K8s集群是否存在异常Pod、特权容器，排查是否有横向渗透的痕迹，重置集群的所有Secret
4. **全盘安全扫描**：对终端、服务器进行全盘扫描，清理后门残留，排查是否有持久化后门

### 6.2 长期缓解措施
1. 依赖版本锁定：在requirements.txt中固定依赖版本，配合`--require-hashes`做哈希校验，避免自动升级到恶意版本
2. CI/CD加固：
   - 引用GitHub Action时，使用固定的Commit SHA，而非可变的版本标签
   - 隔离第三方扫描工具，不赋予其生产发布的高权限凭证
   - 定期轮换CI/CD的凭证，避免凭证长期有效
3. 开发环境最小权限：限制Python进程的系统权限，避免Python进程直接访问敏感目录
4. 供应链监控：部署供应链安全监控，实时监测PyPI包的异常发布、异常变更

## 八、总结与安全建议
### 事件总结
本次LiteLLM投毒事件是TeamPCP团伙的系列供应链攻击中危害最深的一起，攻击者通过阶梯式的链式攻击，攻陷上游安全工具，窃取下游的发布凭证，最终投毒核心AI基础设施。事件暴露了当前开源生态的信任链风险：上游工具的沦陷，会导致大量下游项目无感知中招，同时Python的.pth自动执行机制，也给了攻击者实现"零感知"投毒的机会。

### 安全建议
#### 开发者侧
1. 固定依赖版本，配合哈希校验，避免自动升级
2. 为所有开发、云平台、代码仓库账号开启2FA
3. 定期轮换敏感凭证，避免凭证长期有效
4. 不要在开发环境存储明文的敏感凭证

#### 企业侧
1. 部署供应链安全监控，实时监测依赖的异常变更
2. 加固CI/CD流水线，使用固定的Commit SHA引用Action，定期轮换凭证
3. 隔离开发环境，限制Python进程的系统权限
4. 定期对开发团队做供应链安全的培训，提升安全意识
