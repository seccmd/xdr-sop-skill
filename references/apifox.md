## 一、事件概述
> 2026年3月，国内流行API协作平台Apifox的公网SaaS版桌面客户端遭遇严重供应链投毒攻击。攻击者通过劫持官方CDN域名，篡改客户端动态加载的事件追踪JS文件，利用Electron框架未启用沙箱的安全缺陷，在用户终端执行恶意代码，系统性窃取开发者的SSH密钥、Git凭证、云API密钥等核心敏感资产，攻击持续18天，覆盖全球全平台开发者。

## 三、攻击手法深度分析
### 3.1 初始入侵路径
攻击者通过污染Apifox官方CDN节点，篡改了客户端动态加载的事件追踪脚本`apifox-app-event-tracking.min.js`，实现了供应链投毒的初始入侵：
- 正常的事件追踪脚本大小仅34KB，投毒后被追加了42KB的混淆恶意代码，总大小变为77KB
- 客户端未对该远程JS文件做子资源完整性（SRI）校验，导致文件被篡改后无法感知
- 攻击者使用的恶意C2域名`apifox.it.com`极具迷惑性，极易被误认为官方测试域名，实现社会工程学伪装

### 3.2 恶意代码行为分析
Apifox桌面端基于Electron框架开发，但存在严重的安全配置缺陷：未启用`sandbox`沙箱隔离、开启了`nodeIntegration`暴露Node.js API、关闭了`contextIsolation`上下文隔离，导致渲染进程中的JS代码直接拥有系统级权限。

恶意代码采用多阶段加载的隐蔽攻击流程：
1. **第一阶段：终端指纹采集**
   恶意代码执行后，首先采集主机的MAC地址、CPU信息、主机名、用户名，生成唯一设备指纹，同时窃取Apifox本地存储的登录accessToken，获取用户的身份信息。

2. **第二阶段：C2通信与载荷加载**
   恶意代码主动连接攻击者控制的C2服务器，加密上传设备与用户信息，同时动态加载后续的攻击载荷，加载完成后立即删除DOM中的脚本节点，消除攻击痕迹。

3. **第三阶段：全量敏感信息窃取**
   恶意代码系统性窃取开发者终端的核心敏感资产，分两个版本迭代：
   - v1基础窃取：SSH目录全量私钥/公钥/known_hosts、Shell命令历史（bash/zsh）、Git明文凭据、系统进程列表
   - v2升级窃取：K8s集群配置文件、npm仓库认证Token、Shell环境变量、SVN凭证，同时遍历用户桌面、文档目录，Windows系统额外扫描磁盘目录结构

   所有窃取的数据会经过Gzip压缩+AES-256-GCM加密后上传，完全规避常规DLP检测。

### 3.3 持久化方式
恶意代码具备持久化控制能力，在窃取数据的同时，会植入伪装成"系统遥测服务"的后门进程，实现系统重启后的持久化控制，同时支持通过C2服务器下发任意远程代码，实现横向渗透。

## 四、影响范围评估
### 4.1 受影响版本清单
| 组件名称 | 受影响恶意版本 | 官方安全修复版本 |
|---|---|---|
| Apifox桌面客户端 | <=2.8.18 | >=2.8.19 |

### 4.2 用户影响范围
- 受影响平台：Windows/macOS/Linux 全平台
- 受影响用户：2026年3月4日-3月22日期间使用**公网SaaS版Apifox桌面客户端**的用户
- 不受影响用户：SaaS Web版用户、私有化部署版用户
- 潜在危害：开发者敏感凭证泄露，可导致代码仓库被篡改、云服务器被入侵、企业内网被横向渗透，甚至引发二次供应链攻击

## 五、用户自查与排查方案
### 5.1 版本检查
首先检查本地Apifox的版本，确认是否在风险窗口期内使用过旧版本：
```bash
# 查看Apifox版本，确认是否低于2.8.19
```

### 5.2 恶意残留检查
检查本地Apifox的缓存目录，确认是否存在恶意代码的残留标记：
```bash
# macOS用户
grep -arlE 'rl_mc|rl_headers' ~/Library/Application\ Support/apifox/Local\ Storage/leveldb

# Windows用户（PowerShell）
Select-String -Path $env:APPDATA\apifox\Local Storage\leveldb\* -Pattern 'rl_mc,rl_headers' -List
```
如果输出具体文件，则说明终端已被成功攻击。

### 5.3 网络行为排查
检查系统网络日志，确认是否存在向恶意C2域名的访问记录：
- 检查是否有访问`apifox.it.com`的请求
- 检查是否有访问其他恶意域名的请求（详见IOC指标）

## 六、修复与缓解方案
### 6.1 紧急修复措施
1. **立即升级客户端**：将Apifox升级到官方发布的2.8.19及以上版本，彻底废除动态加载远程JS的机制
2. **全量重置敏感凭证**：
   - 重置所有SSH私钥，同步更新所有服务器的authorized_keys
   - 吊销所有Git平台的Personal Access Token
   - 重置云服务AccessKey、K8s集群Token、数据库密码
   - 清理命令行历史中暴露的所有密码、API密钥
3. **全盘安全扫描**：对终端进行全盘恶意软件扫描，清理后门残留，排查是否存在持久化后门
4. **阻断恶意域名**：在hosts文件或企业防火墙中阻断恶意C2域名，避免后续外联

### 6.2 长期缓解措施
1. 开发工具沙箱化：在沙箱或虚拟机中运行开发工具，限制开发工具的系统权限
2. 开启CDN资源校验：对动态加载的远程资源开启SRI子资源完整性校验
3. 定期凭证轮换：定期轮换开发环境的敏感凭证，避免凭证长期有效
4. 终端行为监控：对开发工具的异常文件访问、网络请求进行监控告警

## 八、总结与安全建议
### 事件总结
本次Apifox投毒事件是一起典型的针对开发工具链的供应链攻击，攻击者利用开发者对官方工具的过度信任，通过攻陷上游CDN供应链，一次性窃取大量开发者的高价值凭证，进而实现大规模的内网渗透。事件暴露了当前开发工具链的安全短板：大量开发工具基于Electron开发，但未做安全配置，同时动态加载远程资源未做完整性校验，给了攻击者可乘之机。

### 安全建议
#### 开发者侧
1. 不要过度信任开发工具，定期检查开发工具的安全更新
2. 为所有开发、代码仓库、云平台账号开启2FA双因素认证
3. 定期轮换敏感凭证，避免凭证长期有效
4. 不要在开发环境存储明文的敏感凭证，考虑在沙箱中运行开发工具

#### 企业侧
1. 部署供应链安全监控系统，实时监测开发工具、依赖的异常变更
2. 隔离开发环境，限制开发工具的系统权限，避免开发工具直接访问核心敏感数据
3. 定期对开发团队做供应链安全的培训，提升安全意识
4. 建立应急响应流程，遇到供应链攻击时可以快速排查、修复
